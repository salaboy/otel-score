OpenTelemetry Maturity Report

Jaeger v2.17.0

Evaluated: 2026-04-23 Chart: jaegertracing/jaeger 4.7.0 Cluster: otel-eval-jaeger (kind) Model: OTel Support Maturity Model for CNCF Projects

Jaeger is a distributed tracing platform for monitoring and troubleshooting microservices, originally developed at Uber and donated to CNCF. Jaeger v2 is a complete rewrite built on top of the OpenTelemetry Collector framework — making it uniquely positioned as both a first-class OTel citizen and a terminal trace backend.

📡 Maturity Profile

Scale

3 — OTel-Optimized: Continuously refined, governed, treated as a long-lived product surface
2 — OTel-Native: OpenTelemetry is the primary interface, designed intentionally for interoperability
1 — OTel-Aligned: OpenTelemetry explicitly supported alongside legacy approaches
0 — Instrumented: Telemetry exists primarily for internal debugging
Overall Profile
Jaeger v2 achieves strong OTel-Native scores across most dimensions, reflecting its architecture as an OTel Collector distribution. Trace Modeling and Multi-Signal are constrained by its role as a terminal trace backend rather than a lack of OTel commitment.

🔬 Dimension Details

1
Integration Surface
2 — OTel-Native ▼

What we observed

Jaeger v2 is itself an OpenTelemetry Collector distribution — OTLP is not bolted on, it is the foundation. Telemetry configuration uses the standard OTel Collector service.telemetry config block with both a Prometheus pull reader (port 8888) and an OTLP push periodic reader. Standard OTEL_TRACES_SAMPLER, OTEL_TRACES_SAMPLER_ARG, and OTEL_EXPORTER_OTLP_* environment variables are all documented and respected. Legacy Jaeger Thrift and Zipkin receiver protocols are maintained for backward compatibility with instrumented services — appropriate for a tracing backend.

What's working well

  • OTLP push (periodic reader) natively supported for metric export
  • Prometheus pull also available — dual-mode integration
  • Standard OTel Collector service.telemetry config block used throughout
  • OTEL_TRACES_SAMPLER, OTEL_EXPORTER_OTLP_* env vars respected
  • Pod annotations enable auto-discovery by Prometheus (prometheus.io/scrape=true)
  • Connects to existing OTel Collectors without adapters or glue code

Opportunities

  • Documenting the integration surface as an explicit stable contract would enable Level 3
  • Providing a migration guide for users moving from v1 legacy exporters to v2 OTLP would reduce upgrade friction
  • Adding an option to export self-traces to an external OTLP endpoint would complete the integration story
2
Semantic Conventions
2 — OTel-Native ▼

What we observed

Jaeger v2 uses current, non-deprecated OTel semantic conventions throughout. Self-trace span attributes include http.request.method, http.response.status_code, url.path, url.scheme, server.address, network.peer.address, and user_agent.original — all current stable semconv. No deprecated attributes (http.method, http.status_code, http.target) were observed. Metrics use http.server.request.duration (not the deprecated form), with data point attributes http.request.method, http.response.status_code, and network.protocol.version. schemaUrl is set consistently: https://opentelemetry.io/schemas/1.38.0 for HTTP metrics, https://opentelemetry.io/schemas/1.18.0 for OTel Collector internal metrics. Instrumentation scope is set: go.opentelemetry.io/contrib/instrumentation/net/http/otelhttp v0.67.0.

What's working well

  • All current stable OTel HTTP semconv used — http.request.method, http.response.status_code, url.path
  • No deprecated attributes observed anywhere in telemetry
  • schemaUrl consistently set on all OTLP metric exports
  • Instrumentation scope name and version set on traces
  • Metric attribute keys align with current semconv

Opportunities

  • Adding OTLP log export would allow full assessment of log semantic conventions
  • Introducing domain-specific jaeger.* attributes for storage operations, query patterns, and sampling decisions would enable Level 3
  • Documenting custom conventions as explicit semantic extensions with versioning would complete the picture
3
Resource Attributes & Configuration
2 — OTel-Native ▼

What we observed

Jaeger natively emits three core identity attributes: service.name: jaeger, service.version: v2.17.0, and service.instance.id: <UUID> (stable per process lifecycle). These are set in service.telemetry.resource and appear consistently across metrics (OTLP push), metrics (Prometheus scrape), self-traces (internal), and logs (stdout JSON). Standard OTEL_TRACES_SAMPLER, OTEL_TRACES_SAMPLER_ARG, and OTEL_EXPORTER_OTLP_* env vars are all documented and respected. Kubernetes resource attributes (k8s.pod.name, k8s.namespace.name, etc.) are not emitted natively — they are correctly pipeline-derived via the k8sattributes processor.

What's working well

  • service.name, service.version, service.instance.id set natively
  • Identity is consistent across all signals (metrics, traces, logs)
  • OTEL_TRACES_SAMPLER and OTEL_EXPORTER_OTLP_* respected end-to-end
  • Identity attributes placed on resource, not on spans
  • Delegates to OTel Collector SDK for configuration — correct approach

Opportunities

  • Explicitly documenting configuration precedence (service.telemetry.resource vs OTEL_SERVICE_NAME) would reduce user confusion
  • Providing guidance for multi-tenant deployments (per-namespace vs per-instance identity) would enable Level 3
  • Treating identity changes as breaking changes with migration guidance would strengthen stability
4
Trace Modeling & Context Propagation
1 — OTel-Aligned ▼

What we observed

Jaeger v2.17.0 implements safe self-tracing for its Query service (PR #8208), covering HTTP operations like /api/traces and /api/services. Self-traces are stored internally in Jaeger's own storage backend (not exported to external OTLP collectors). Span kind is correctly set to server. The default sampler is parentbased_always_on — Jaeger traces its own requests when they arrive with a traceparent header. A recursive self-tracing loop prevention mechanism is in place. Jaeger accepts trace context on all receiver protocols: W3C TraceContext (OTLP, verified HTTP 200), B3 (Zipkin, verified HTTP 202), and Jaeger format (Thrift/gRPC, verified port active). As a terminal trace backend, Jaeger does not propagate context outbound — this is by design.

What's working well

  • W3C TraceContext, B3, and Jaeger propagation formats all accepted
  • Self-traces use correct server span kind
  • Self-tracing loop prevention implemented (non-trivial for a trace backend)
  • OTEL_TRACES_SAMPLER=always_off documented for production tuning
  • Default parentbased_always_on sampler is sensible

Opportunities

  • Providing an option to export self-traces to an external OTLP endpoint would enable monitoring Jaeger in a shared observability stack
  • Improving span names from URL paths (/api/traces) to logical operation names (GET /api/traces) would improve usability
  • Expanding self-tracing to storage operations (jaeger_storage extension) would make Jaeger's own performance observable
  • Adding child spans for internal operations (e.g., storage lookups) would create richer trace trees
5
Multi-Signal Observability
1 — OTel-Aligned ▼

What we observed

Metrics are the only signal that flows to external OTel pipelines — both via OTLP push (periodic reader, 15s) and Prometheus pull (port 8888). 23 native metric names observed, covering pipeline throughput (otelcol_receiver_accepted_spans, otelcol_exporter_sent_spans), HTTP server performance (http.server.request.duration), and process health. Self-traces are stored internally in Jaeger's own storage — not available in shared observability backends. Logs go to stdout as structured JSON with embedded OTel resource fields but are not exported via OTLP and do not include trace_id or span_id for correlation. No cross-signal correlation is possible in standard OTel tooling with the current configuration.

What's working well

  • Metrics are first-class with dual OTLP + Prometheus export
  • Self-traces exist and cover Query service operations
  • Logs are structured JSON with OTel resource fields embedded
  • All signals share consistent service.name identity

Opportunities

  • Adding OTLP log export would enable cross-signal correlation in shared backends
  • Including trace_id and span_id in structured log output would link log events to self-traces
  • Providing an option to export self-traces externally would unify all three signals in a single observability platform
  • Documenting how to correlate Jaeger's own telemetry signals would help operators set up holistic monitoring

Audience & Signal Quality
2 — OTel-Native ▼

What we observed

Metrics are operationally focused and well-documented. The official monitoring guide explicitly calls out the 4 key metrics to watch: otelcol_receiver_accepted_spans, otelcol_receiver_refused_spans, otelcol_exporter_sent_spans, and otelcol_exporter_send_failed_spans. Metrics verbosity is configurable via metrics.level (basic, normal, detailed). Self-trace span names are URL paths (/api/traces, /api/services) — functional but not prefixed with HTTP method. Span kinds are correctly set. A Grafana dashboard (code-generated via ADR-007) and Prometheus alert rules are provided in the monitoring mixin — however, the alert rules still reference v1 metric names (jaeger_agent_*, jaeger_collector_*) that do not exist in v2.

What's working well

  • Key metrics explicitly documented in the monitoring guide
  • Metrics level is configurable (basic / normal / detailed)
  • Grafana dashboard provided and kept in sync with code generation (ADR-007)
  • No internal function names in span names
  • Correct span kinds (server) on self-traces
  • OTEL_TRACES_SAMPLER=always_off documented for production noise reduction

Opportunities

  • Update Prometheus alert rules in the monitoring mixin — current rules use v1 metric names (jaeger_agent_*) that don't exist in v2
  • Prefixing self-trace span names with HTTP method (e.g., GET /api/traces) would improve usability
  • Expanding self-tracing to storage operations would give operators visibility into Jaeger's own latency
  • Adding formal metrics snapshot tests would enable Level 3 quality regression detection

Stability & Change Management
2 — OTel-Native ▼

What we observed

Telemetry-related changes are documented in release notes with PR links. v2.17.0 includes entries for "Safe self-tracing via otel collector telemetry factory (#8208)" and "Implement timer duration bucket parsing in metrics init (#7951)". A CI workflow generates metric change diffs on PRs ("Include metric change details in pr comment" #8153) — a proactive stability mechanism. schemaUrl is consistently set on all OTLP metric exports (1.38.0 for HTTP metrics, 1.18.0 for OTel Collector internals). The Grafana dashboard is code-generated from Go source using grafana-foundation-sdk, keeping it in sync with metric changes. The monitoring mixin's Prometheus alert rules still reference v1 metric names (jaeger_agent_*, jaeger_reporter_*) — these are broken for v2 and represent a stability gap for users relying on the official monitoring tooling.

What's working well

  • Telemetry changes documented in release notes with PR references
  • schemaUrl consistently set on all OTLP metric exports
  • CI metric change detection on PRs (proactive)
  • Grafana dashboard kept in sync via code generation (ADR-007)
  • Monitoring documentation explicitly describes telemetry behavior
  • Self-tracing behavior and disable mechanism documented

Opportunities

  • Update Prometheus alert rules in the mixin to use v2 metric names (otelcol_receiver_*, otelcol_exporter_*)
  • Adding an explicit stability policy (e.g., "metrics with otelcol_ prefix are stable") would help users build reliable dashboards
  • Providing a v1→v2 migration guide for telemetry (metric names, alert rules) would reduce upgrade friction
  • Distinguishing stable vs experimental telemetry in documentation would enable Level 3

💡 Key Takeaways

✅ Strengths

  • Architecture-level OTel commitment — Jaeger v2 IS an OTel Collector distribution; OpenTelemetry is the foundation, not an add-on
  • Current semantic conventions — http.request.method, http.response.status_code, url.path used throughout; no deprecated attributes
  • Dual metric export — OTLP push + Prometheus pull natively, compatible with both modern and legacy stacks
  • Schema URLs consistently set — enables schema-aware tooling and version tracking
  • Safe self-tracing — v2.17.0 implements recursive loop prevention for a trace backend that traces itself
  • CI metric change detection — PR comments include metric diffs; metrics treated as a contract
  • Stable resource identity — service.name, service.version, service.instance.id consistent across all signals

🎯 Priority Improvements

  • Update Prometheus alert rules — the monitoring mixin still uses v1 metric names that don't exist in v2; this breaks official alerting guidance
  • Add OTLP log export — would enable cross-signal correlation and complete the multi-signal picture
  • Enable external self-trace export — allow users to monitor Jaeger's own behavior in their shared observability stack
  • Add trace context to logs — include trace_id/span_id in structured log output for correlation
  • Expand self-tracing coverage — add storage operation spans to make Jaeger's own latency observable

🔍 Notable Observations

  • Unique architectural position — as both an OTel Collector distribution and a terminal trace backend, some dimensions are constrained by role rather than OTel commitment
  • Outdated TODO comment — the default config.yaml still says "TODO Initialize telemetry tracer" but v2.17.0 actually implements self-tracing
  • Code-generated Grafana dashboard — ADR-007 establishes a workflow to regenerate the dashboard from Go source, keeping it in sync with metric changes
  • parentbased_always_on default sampler — Jaeger traces its own requests only when they arrive with a traceparent header; avoids noise while enabling observability
