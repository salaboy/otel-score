1. Integration Surface

OTLP HTTP traces + Prometheus metrics; dual config systems; no OTLP logs
Native
⌄
Key evidence: Traces export via OTLP HTTP to :4318/v1/traces configured in the config-tracing ConfigMap. Metrics exposed as Prometheus at :9090 on controller, webhook, and events-controller services. The config-observability ConfigMap documents metrics-protocol: grpc and http/protobuf options for OTLP push, suggesting that path is in development. Dynamic config reload confirmed — no controller restart required.

What's Working Well

  • Traces export via native OTLP HTTP — no sidecar or proxy needed
  • Dynamic config reload — endpoint changes take effect without restart
  • OTel SDK (v1.42.0) used natively for both traces and metrics
  • Prometheus metrics at :9090 on all 3 controller services
  • ConfigMap-based configuration is Kubernetes-native and auditable

Opportunities to Improve

  • Add OTLP log export capability — currently logs go to stdout only with no OTel path
  • Consolidate dual tracing config systems: deprecate config-tracing in favor of config-observability tracing keys
  • Confirm and document OTLP metrics push (metrics-protocol: grpc/http) as a supported path
  • Add support for gRPC OTLP traces (currently HTTP only)

2. Semantic Conventions

OTel SDK used natively; HTTP metrics follow current semconv; trace span attributes use custom keys
Native
⌄
HTTP metrics from the otelhttp instrumentation library correctly use current semconv: http.request.method, http.response.status_code, network.protocol.name, server.address. Schema URL present: traces at 1.12.0, metrics at 1.18.0. Span attributes use custom keys: taskrun, namespace, pipelinerun — rather than OTel semconv equivalents like k8s.taskrun.name or k8s.namespace.name. No error status codes set on any spans.

What's Working Well

  • HTTP metrics use current OTel semconv — http.request.method not deprecated http.method
  • Schema URL present on both traces and metrics
  • Go runtime metrics use OTel community conventions (go_*)
  • Knative workqueue metrics use consistent kn_workqueue_* prefix
  • Span kinds are correct (INTERNAL for reconciler operations)

Opportunities to Improve

  • Update trace span attributes from taskrun/namespace to k8s.taskrun.name/k8s.namespace.name
  • Set STATUS_CODE_ERROR on spans when reconciliations fail — currently all spans are UNSET
  • Update trace schema URL from outdated 1.12.0 to current version (1.28.0+)
  • Add version to instrumentation scope names (TaskRunReconciler currently has no version)
  • Align Tekton metric attribute keys with semconv where applicable (e.g., k8s.namespace.name)

3. Resource Attributes & Configuration

service.name set natively; no service.version in traces; identity inconsistent across signals
Aligned
⌄
Key issue — identity inconsistency: Traces report service.name = taskrun-reconciler / pipelinerun-reconciler, while metrics report service.name = tekton-pipelines-controller for the same controller process. Cross-signal correlation by service name is impossible. service.version is absent from traces; metrics carry service_version: 5a88281 (git commit hash in underscore form). No OTEL_* environment variables found in the controller deployment.

What's Working Well

  • service.name set natively in OTLP trace exports
  • OTel SDK identity attributes visible in metrics (telemetry_sdk_name: opentelemetry, telemetry_sdk_version: 1.42.0)
  • ConfigMap-based configuration is easy to audit and version-control

Opportunities to Improve

  • Align service.name between traces and metrics — use the same value for the same process
  • Add service.version: v1.11.1 to trace resource attributes (not just git commit hash)
  • Add service.instance.id to trace resource attributes
  • Support standard OTEL_SERVICE_NAME and OTEL_RESOURCE_ATTRIBUTES env vars for operator overrides
  • Use dot-notation for resource attribute keys in Prometheus labels (service.version not service_version)

4. Trace Modeling & Context Propagation

Excellent: cross-service traces, W3C propagation to TaskRun pods, deep reconciler span trees
Optimized
⌄
Cross-service trace confirmed: A single PipelineRun trace (bc5bc7d6...) contains spans from both pipelinerun-reconciler and taskrun-reconciler services — the createTaskRun span is the parent of the TaskRun reconciler's root. W3C propagation to workloads: Tekton injects tekton.dev/taskrunSpanContext: {"traceparent":"00-bc5bc7d6...-01"} into TaskRun annotations, enabling user pipeline code to join the parent trace. Span events present on root spans.

What's Working Well

  • Cross-service traces span both pipelinerun-reconciler and taskrun-reconciler
  • W3C traceparent propagated to TaskRun pods via tekton.dev/taskrunSpanContext annotation
  • Deep, logical span hierarchy: TaskRun:Reconciler → prepare → createPod → reconcile → ...
  • 17 unique span names covering the full reconciler lifecycle
  • Span events on root spans (updating TaskRun status with SpanContext)
  • Correct span kinds (INTERNAL) for controller operations

Opportunities to Improve

  • Set STATUS_CODE_ERROR on spans when reconciliations fail — currently all spans are UNSET even for failed TaskRuns
  • Add richer span attributes: k8s.taskrun.name, k8s.namespace.name, k8s.pipeline.name using semconv keys
  • Document the span catalog (names, attributes, hierarchy) so operators know what to expect

5. Multi-Signal Observability

Two signals flowing; logs stdout-only; cross-signal correlation broken by service.name mismatch
Native
⌄
Signal status: Traces (OTLP HTTP) and Metrics (Prometheus) both flowing. Logs go to stdout as structured JSON — no OTLP export. Correlation gap: Traces use service.name = taskrun-reconciler while metrics use service.name = tekton-pipelines-controller for the same process — making trace-metric correlation by service name impossible. Log knative.dev/traceid uses UUID format (e.g., 5c3888ed-4346-4f32-8dad-e1e747f85a83), incompatible with OTel hex trace IDs.

What's Working Well

  • Two signals flowing with meaningful content
  • Tekton-specific metrics cover core CI/CD dimensions: duration, counts, running, waiting
  • OTel SDK used internally for both traces and metrics
  • 11 Tekton-specific metrics plus Go runtime, Knative infra, and HTTP metrics

Opportunities to Improve

  • Unify service.name across traces and metrics to enable cross-signal correlation
  • Add OTLP log export — or at minimum replace knative.dev/traceid (UUID) with the OTel trace ID (hex) in stdout logs
  • Consider OTLP metrics push as an alternative to Prometheus scrape for simpler collector configuration

6. Audience & Signal Quality

Operationally useful traces and metrics; no error status codes on failed runs
Native
⌄
Span names are operationally meaningful: TaskRun:Reconciler, prepare, createPod, resolvePipelineState, runNextSchedulableTask. Most are understandable without source code access. Critical gap: Failed TaskRuns (e.g., fail-taskrun-1) produce no error status codes — all 2,217 spans are STATUS_CODE_UNSET. Failures are only visible in metrics (tekton_pipelines_controller_taskrun_total{status="failed"}), not in traces.

What's Working Well

  • Span names reflect logical reconciler operations, not just function names
  • Metrics cover key CI/CD questions: duration histograms, run counts, queue depth
  • Cross-service traces give end-to-end pipeline execution visibility
  • Metric documentation exists at docs/metrics.md with types and labels
  • Low signal-to-noise ratio — 17 span types, all relevant

Opportunities to Improve

  • Set STATUS_CODE_ERROR on reconciler spans when TaskRun/PipelineRun fails — currently failures are invisible in traces
  • Add span attributes with more operational context: failure reason, step name, exit code
  • Publish a trace span catalog in documentation alongside the metrics reference
  • Provide sample Grafana dashboards or Prometheus alert rules in the repository
  • Clarify and consolidate the dual tracing config systems for operators

7. Stability & Change Management

Metrics documented as "experimental"; exemplary migration guide; no trace stability guarantees
Native
⌄
Strengths: docs/metrics.md lists all metrics with types, labels, and explicit "experimental" status. docs/metrics-migration-otel.md provides a detailed OpenCensus → OTel migration guide with breaking change tables and a migration checklist — an exemplary change communication document. Schema URL present on both signals. Gaps: All metrics marked "experimental" (no stability guarantee). No equivalent trace documentation. Trace schema URL is outdated (1.12.0 from 2021).

What's Working Well

  • Metrics documented with names, types, labels, and status in docs/metrics.md
  • Breaking changes communicated via dedicated migration guide with before/after tables
  • Schema URL present on both trace and metric exports
  • OTel SDK version (1.42.0) is current and actively maintained
  • ConfigMap examples serve as inline configuration reference

Opportunities to Improve

  • Graduate core metrics from "experimental" to stable — provide stability guarantees for taskrun_total, pipelinerun_total, duration histograms
  • Publish a trace span catalog (span names, attributes, hierarchy) alongside metrics docs
  • Update trace schema URL from 1.12.0 to current (1.28.0+)
  • Explicitly deprecate config-tracing ConfigMap in favor of config-observability tracing keys
💡 Key Takeaways

✦ Top Strengths

  • Best-in-class trace modeling: Cross-service traces span PipelineRun + TaskRun reconcilers, with W3C traceparent propagated to user workload pods via annotations — enabling distributed tracing into user CI/CD code
  • Native OTel SDK usage: Go OTel SDK v1.42.0 used for both traces and metrics; HTTP metrics follow current semantic conventions; schema URL present on both signals
  • Exemplary migration communication: The OpenCensus → OTel migration guide in docs/metrics-migration-otel.md with explicit breaking change tables is a model for the CNCF ecosystem

↑ Priority Improvements

  • Fix service.name inconsistency: Traces report taskrun-reconciler while metrics report tekton-pipelines-controller for the same process — preventing cross-signal correlation in any observability platform
  • Add error status codes to spans: All spans are STATUS_CODE_UNSET even for failed TaskRuns — operators cannot use trace status to detect failures, only metrics
  • Enable log-trace correlation: Replace internal knative.dev/traceid (UUID) with OTel trace ID (hex) in stdout logs, or add OTLP log export

◈ Notable Observations

  • Dual tracing systems: v1.11.1 has two parallel tracing config paths — legacy config-tracing (working) and new config-observability tracing keys (documented but not yet effective) — creating operator confusion
  • Intentional service.name divergence: Traces use reconciler role names, metrics use deployment names — appears deliberate but breaks cross-signal correlation
  • Prometheus underscore notation: Resource attributes from Prometheus scrape use service_version (underscore) alongside service.version (dot) — a Prometheus-to-OTel translation artifact resulting in duplicate attributes
Methodology: Telemetry collected using OpenTelemetry Collector v0.150.1 with file export in a kind cluster (otel-eval-tekton). Traffic generated via 6 TaskRuns, 5 PipelineRuns (10 child TaskRuns), and 1 failing TaskRun. The k8sattributes processor was used to distinguish project-native vs collector-enriched resource attributes. Semantic conventions checked against the latest stable OpenTelemetry specification.
