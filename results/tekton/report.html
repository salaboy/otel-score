Dimension Details
2 Integration Surface OTLP HTTP traces + Prometheus metrics ▶

What we observed

Tekton exports traces via OTLP HTTP (otlptracehttp) to any OTLP-compatible endpoint. Metrics are exposed via a Prometheus scrape endpoint at :9090 on the controller, webhook, and events-controller services. Logs remain on stdout only.

Configuration uses two Kubernetes ConfigMaps: config-tracing (for OTLP traces) and config-observability (for metrics). No OTEL_* environment variable support is available.

What's working well

  • OTLP HTTP traces confirmed flowing — 519 spans captured across TaskRun and PipelineRun reconciliation
  • Prometheus metrics available by default (set in the release manifest as metrics-protocol: prometheus)
  • Three separate Prometheus endpoints for fine-grained scraping
  • The config-observability ConfigMap documents grpc and http/protobuf as future OTLP metrics options

Opportunities

Enable tracing by default — Tracing requires manual ConfigMap configuration. Providing a default OTLP endpoint (e.g., http://localhost:4318/v1/traces) or enabling traces by default would improve adoption.
Add OTEL_* environment variable support — Supporting OTEL_EXPORTER_OTLP_ENDPOINT and OTEL_SERVICE_NAME would allow standard tooling (Helm charts, Kubernetes operators) to configure Tekton's telemetry without Kubernetes-specific ConfigMap patching.
Add OTLP log export — Structured logs are already emitted to stdout as JSON. Routing them via OTLP would complete the three-signal story.

Checklist

  • ✅ OTLP traces confirmed flowing
  • ✅ Prometheus metrics endpoint documented and available
  • ❌ No OTLP metrics push
  • ❌ No OTLP log export
  • ❌ No OTEL_* environment variable support
1 Semantic Conventions Custom naming; schema URL present but outdated ▶

What we observed

Span attributes use custom keys: taskrun, namespace, pipelinerun. The OTel CI/CD semantic conventions (e.g., cicd.pipeline.run.id, cicd.task.run.id) are not used. The schema URL is set to https://opentelemetry.io/schemas/1.12.0 — present but outdated (current: 1.29.0+).

Metrics use Prometheus naming convention with _ separators (tekton_pipelines_controller_taskrun_duration_seconds) rather than OTel dot-notation. Metric attribute keys like pipeline, task, status, reason are domain-meaningful but not aligned with OTel conventions.

The instrumentation scope is named TaskRunReconciler / PipelineRunReconciler but has no version set.

What's working well

  • service.name is correctly set using semconv.ServiceNameKey
  • Schema URL is present on trace exports (showing awareness of OTel versioning)
  • Span names are semantically meaningful and describe operations clearly
  • Metric labels (pipeline, task, status, reason) are useful for CI/CD dashboards

Opportunities

Adopt OTel CI/CD semantic conventions — Replace taskrun → cicd.task.run.id, pipelinerun → cicd.pipeline.run.id, namespace → k8s.namespace.name. This would enable correlation with other CI/CD tools using the same conventions.
Update schema URL — Bump from semconv/v1.12.0 to the current semconv/v1.29.0 import. This is a low-effort change that signals commitment to keeping up with the spec.
Set instrumentation scope version — Add the Tekton release version to the scope name in createTracerProvider() to enable version-based filtering in trace backends.
1 Resource Attributes & Configuration Only service.name native in traces; inconsistent identity across signals ▶

What we observed

Traces natively emit only service.name (taskrun-reconciler or pipelinerun-reconciler). No service.version, service.instance.id, or telemetry.sdk.* attributes are set natively.

Prometheus metrics (via Knative) include richer attributes: service.name, service.instance.id, telemetry_sdk_language: go, telemetry_sdk_name: opentelemetry, telemetry_sdk_version: 1.42.0. However, service_version contains a git commit hash (5a88281), not the release version (v1.11.1).

Critical identity gap: traces use service.name = taskrun-reconciler while metrics use service.name = tekton-pipelines-controller. Cross-signal correlation by service name is not possible without additional enrichment.

What's working well

  • service.name is set in both traces and metrics (different values, but present)
  • Prometheus metrics include full SDK identity attributes via the Knative framework
  • The k8sattributes processor enriches traces with rich Kubernetes identity (deployment, pod, node)

Opportunities

Unify service.name across signals — Use tekton-pipelines-controller (matching the Kubernetes Deployment) in both traces and metrics. This enables correlation in observability backends like Grafana, Honeycomb, or Datadog.
Add service.version to traces — Set semconv.ServiceVersionKey.String("v1.11.1") in createTracerProvider(). This enables version-based filtering and regression analysis.
Use release version in service_version — Replace the git commit hash with the actual release tag (v1.11.1) for the Prometheus service_version attribute.
2 Trace Modeling & Context Propagation Rich reconciliation spans; W3C traceparent injected into workload pods ▶

What we observed

16 unique span names covering the complete reconciliation lifecycle were observed. Parent-child relationships are well-formed: 518 of 519 spans have a parent span ID. W3C Trace Context is propagated to TaskRun pods via the tekton.dev/taskrunSpanContext annotation:

{"traceparent":"00-3dbb099caf71396db5596e9a20f9fc6f-ee013ea73bfc5df8-01"}

This allows workloads running inside TaskRuns to join the parent reconciler trace — a powerful integration point for CI/CD observability.

However, all 519 spans have status=UNSET. Even the failed TaskRun (fail-taskrun) produced no ERROR-status spans, making trace-based error alerting impossible. PipelineRun and TaskRun traces are also disconnected (different service.name values).

What's working well

  • 16 meaningful span names: TaskRun:ReconcileKind, prepare, reconcile, createPod, resolvePipelineState, runNextSchedulableTask, etc.
  • W3C Trace Context propagated to workload pods — workloads can attach to the CI/CD trace
  • propagation.TraceContext{} propagator configured globally
  • Proper span hierarchy: root spans with meaningful child spans

Opportunities

Set ERROR status on failed spans — When a TaskRun fails, set span.SetStatus(codes.Error, "taskrun failed: reason"). This enables trace-based alerting and error rate calculation in observability backends.
Link PipelineRun and TaskRun traces — Use a trace link or shared service.name to connect PipelineRun traces with their child TaskRun traces. Currently they are completely separate trace trees.
Accept incoming trace context — Allow external systems (CI triggers, webhooks) to inject a parent span ID into PipelineRun creation, enabling end-to-end tracing from trigger to completion.
2 Multi-Signal Observability Traces + Metrics; no OTLP logs; no cross-signal correlation ▶

What we observed

Two signals are flowing: traces via OTLP HTTP and metrics via Prometheus scrape. Logs are emitted to stdout as structured JSON but are not exported via OTLP.

Cross-signal correlation is not possible: traces use service.name = taskrun-reconciler while metrics use service.name = tekton-pipelines-controller. Logs include a knative.dev/traceid field but this is an internal UUID, not an OTel trace ID. No exemplars link Prometheus metrics to traces.

What's working well

  • Both traces and metrics are first-class signals with documentation
  • 19 Tekton-specific metrics directly reflect CI/CD domain concepts
  • Traces cover both TaskRun and PipelineRun reconciliation paths
  • Prometheus metrics include 77 infrastructure metrics (Go runtime, Knative workqueue, HTTP client)

Opportunities

Add exemplars to Prometheus metrics — Linking tekton_pipelines_controller_taskrun_duration_seconds to trace IDs via exemplars would enable "jump to trace" from Grafana dashboards.
Export logs via OTLP — Structured JSON logs are already emitted. Routing them to the OTLP log pipeline would complete the three-signal story and enable log-trace correlation.
Unify service.name — Using a consistent service.name across traces and metrics is the single highest-impact change for enabling cross-signal correlation.
2 Audience & Signal Quality CI/CD-domain metrics; meaningful spans; all metrics labeled "experimental" ▶

What we observed

Span names are operator-meaningful: TaskRun:ReconcileKind, prepare, createPod, resolvePipelineState. Each reconciliation cycle produces ~7–10 spans — appropriate granularity for debugging without excessive verbosity.

Metrics are domain-specific: tekton_pipelines_controller_pipelinerun_duration_seconds, tekton_pipelines_controller_taskrun_total, tekton_pipelines_controller_running_pipelineruns. These directly answer CI/CD operational questions. However, all metrics are labeled "experimental" in the documentation — no stability commitment.

Error observability gap: failed TaskRuns produce no ERROR-status spans, making it impossible to use traces for error alerting without additional instrumentation.

What's working well

  • Metrics cover CI/CD lifecycle: duration, counts, running state, pod latency
  • Span names describe operations, not internal code paths
  • Metrics are available by default (no configuration required)
  • Metric labels (pipeline, task, status, reason) enable useful dashboards
  • Documentation warns about tekton_pipelines_controller_taskruns_pod_latency_milliseconds unbounded cardinality (issue #9393) — good transparency

Opportunities

Set ERROR span status on failures — When a TaskRun step exits with non-zero, set span.SetStatus(codes.Error, msg) and span.RecordError(err). This is the highest-impact quality improvement for traces.
Graduate metrics from "experimental" — Committing to stability for the core tekton_pipelines_controller_pipelinerun_* and tekton_pipelines_controller_taskrun_* metrics would give operators confidence to build SLOs on them.
Enable tracing by default — Tracing is the most powerful signal for CI/CD debugging but requires manual configuration. A default OTLP endpoint or a simpler one-step setup would improve adoption.
2 Stability & Change Management Migration guide for breaking changes; schema URL present but outdated ▶

What we observed

Tekton has dedicated observability documentation: docs/metrics.md (complete metric reference), docs/metrics-migration-otel.md (breaking change migration guide for OpenCensus→OTel), and docs/developers/tracing.md (tracing setup). The migration guide is thorough, documenting old vs new metric names and a migration checklist.

Schema URL is set to https://opentelemetry.io/schemas/1.12.0 on traces — present but outdated (current: 1.29.0+). All metrics are explicitly labeled "experimental" in the documentation, meaning no stability guarantee is made.

What's working well

  • Dedicated metrics reference documentation with complete table
  • Thorough migration guide for the OpenCensus→OpenTelemetry transition (PR #9043)
  • Core Tekton metrics preserved for backward compatibility during migration
  • Schema URL present on trace exports
  • Transparency about cardinality risks in the documentation

Opportunities

Graduate core metrics from "experimental" — The tekton_pipelines_controller_pipelinerun_* and tekton_pipelines_controller_taskrun_* metrics are stable in practice. Removing the "experimental" label would signal a stability commitment.
Update schema URL to current version — Bump the semconv import from v1.12.0 to v1.29.0. This is a low-effort change that keeps the schema URL current.
Update tracing docs to be OTLP-generic — The tracing documentation references Jaeger specifically. Updating it to reference any OTLP HTTP endpoint (with Jaeger as an example) would be more accurate and broadly applicable.
Key Takeaways

🟢 Strengths

  • W3C Trace Context propagation to workloads — Tekton injects tekton.dev/taskrunSpanContext into TaskRun pods, enabling workloads to join the CI/CD trace
  • Rich metrics with migration documentation — 19 domain-specific metrics with a thorough OpenCensus→OTel migration guide
  • Meaningful reconciliation traces — 16 span names covering the complete lifecycle with proper parent-child relationships

🟡 Priority Improvements

  • Set ERROR status on failed spans — Highest-impact trace quality fix
  • Unify service.name across signals — Enables cross-signal correlation in observability backends
  • Add service.version to traces — Enables version-based filtering and regression analysis
  • Enable tracing by default — Reduces adoption friction

🔵 Notable Observations

  • Two tracing systems in v1.11.1 — The legacy config-tracing ConfigMap and the newer config-observability tracing keys coexist, creating potential operator confusion
  • OTel SDK used internally for Prometheus metrics — The otel_scope_name label in Prometheus output reveals OTel SDK usage even for Prometheus-exposed metrics
  • Tekton pod labels enable collector-side correlation — Labels like tekton.dev/taskRun and tekton.dev/pipelineRun on pods allow the k8sattributes processor to enrich traces with CI/CD context
